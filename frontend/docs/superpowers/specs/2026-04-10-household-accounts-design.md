# Household Accounts — Design Spec

**Issue:** #1 Split Household Accounts  
**Date:** 2026-04-10  
**Status:** Approved

## Overview

Households are the fundamental unit of PlatePlan. Every user belongs to one or more households. A household has one shared meal plan generated by merging all members' preferences. There is no "solo mode" — a user who wants a personal plan simply creates a single-member household.

Users can belong to multiple households (e.g. one for lunch with flatmates, one for dinner with a partner). One household is "active" at a time, driving the meal plan and shopping list views. Switching is done from the Profile tab.

---

## 1. Data Model

### New Tables

**`households`**
```
id            uuid PK
name          text NOT NULL
created_by    uuid FK → auth.users
created_at    timestamptz
updated_at    timestamptz
```

**`household_members`**
```
id             uuid PK
household_id   uuid FK → households
user_id        uuid FK → auth.users
role           text CHECK IN ('owner', 'member')
joined_at      timestamptz
UNIQUE (household_id, user_id)
```

**`household_invites`**
```
id             uuid PK
household_id   uuid FK → households
token          text UNIQUE NOT NULL   -- random UUID, used in deep link
created_by     uuid FK → auth.users
expires_at     timestamptz            -- 7 days from creation
created_at     timestamptz
```

**`household_settings`**  
Stored as columns on `households` (not a separate table):
```
managed_meal_slots    text[]     -- which slots this household plans together
shopping_days         int[]      -- 0=Sun … 6=Sat
batch_cook_days       int        -- 1 | 2 | 3
```

### Modified Tables

| Table | Change |
|---|---|
| `meal_plans` | `user_id` → `household_id` (uuid FK → households) |
| `shopping_lists` | `user_id` → `household_id` (uuid FK → households) |

### Unchanged Tables

`user_preferences`, `recipes`, `planned_meals`, `meal_feedback`, `receipt_items`, `user_favorites`, `automations` — all untouched.

### RLS Policies

| Table | Policy |
|---|---|
| `households` | Members can SELECT their own households; owner can UPDATE |
| `household_members` | Readable by all members of the same household; owner can DELETE members |
| `household_invites` | Any authenticated user can SELECT by token (for join validation); members can INSERT |
| `meal_plans` | SELECT/INSERT/UPDATE/DELETE if user is in `household_members` for that `household_id` |
| `shopping_lists` | Same as `meal_plans` |

## 2. Active Household State

**Store:** `householdStore.ts` (Zustand + AsyncStorage) — same pattern as `onboardingStore.ts`

```typescript
interface HouseholdStore {
  activeHouseholdId: string | null;
  setActiveHouseholdId: (id: string) => void;
  pendingInviteToken: string | null;      // stored when deep link fires before auth
  setPendingInviteToken: (token: string | null) => void;
}
```

**React Query cache keys:** All hooks (`usePlan`, `useShoppingList`) use `activeHouseholdId` in their query keys. Switching households is instant if already cached.

**Bootstrap logic in `_layout.tsx`:**
1. Check Supabase session → not authenticated → `(auth)/welcome`
2. Check `hasCompletedOnboarding` → false → `(onboarding)/step-goals`
3. Fetch user's households → none → `(onboarding)/step-household`
4. Validate `activeHouseholdId` still in membership list → if stale, default to first household
5. Enter `(tabs)`

**Meal plan tab header:** Displays the active household's name below the screen title.

## 3. Invite Link Flow

### Generating

1. Member taps "Share invite link" in Profile → Household detail
2. `create-invite` Edge Function: inserts `household_invites` row with `crypto.randomUUID()` token + 7-day expiry, returns token
3. App builds link: `plateplan://invite/<token>`
4. `Share.share()` opens native share sheet (WhatsApp, SMS, iMessage, etc.)

### Accepting

1. Recipient opens link → Expo deep link handler in `_layout.tsx` extracts token
2. If not authenticated: token saved to `pendingInviteToken` in `householdStore`, user sent to sign-up/onboarding; token processed after auth completes
3. If authenticated: `join-household` Edge Function called immediately
4. Edge Function validates: token exists, not expired, user not already a member → inserts `household_members` row
5. On success: `activeHouseholdId` set to joined household, routed to `(tabs)`
6. On failure: Alert with clear message (expired / already a member)

## 4. Onboarding Change

**New final step:** `(onboarding)/step-household.tsx` inserted after `step-complete`

**Two paths:**

**Create:** Text input for household name → taps "Create" → `create-household` Edge Function (creates household + adds creator as owner) → `activeHouseholdId` set → marks `hasCompletedOnboarding = true` → enters `(tabs)`

**Join:** If `pendingInviteToken` is set in `householdStore`, the screen pre-selects the join path and shows the household name being joined → taps "Join" → `join-household` Edge Function → `activeHouseholdId` set → marks `hasCompletedOnboarding = true` → enters `(tabs)`

**Existing fully-onboarded users accepting an invite:** Deep link bypasses onboarding entirely — `join-household` is called directly, `activeHouseholdId` switches to the new household.

## 5. Plan Generation — Preference Merging

`generate-plan` Edge Function signature changes:

```typescript
// Before
{ userId: string; weekStart: string; preferences: UserPreferences }

// After
{ householdId: string; weekStart: string }
```

The function fetches all members' `user_preferences` internally, then merges:

| Field | Merge strategy |
|---|---|
| `dietaryRestrictions` | Union — most restrictive wins |
| `likedIngredients` | Union |
| `dislikedIngredients` | Union |
| `likedCuisines` | Union |
| `managedMealSlots` | From `household_settings` |
| `shoppingDays` | From `household_settings` |
| `batchCookDays` | From `household_settings` |
| `maxCookTimeMinutes` | Minimum across members |
| `seasonalityImportance` | Average, rounded |
| `pantryStaples` | Union |
| `calorieTarget` | Summed (plan feeds all members) |
| `proteinTargetG` | Summed |

Claude is told the household member count; recipe `servings` are expected to match.

The same merge logic applies to `regenerate-meal`.

`generate-shopping-list` already works off `plan_id` so its signature is unchanged; it just needs the RLS update so members can access the plan.

## 6. UI Changes

### Profile Tab

New "Households" section at the top (above Nutrition targets):

- Lists all households with name, member count, active checkmark
- Tap row → switches `activeHouseholdId`
- Tap chevron → Household detail screen (members list, settings, share invite, leave)
- "Create new household" button at section bottom

**Household detail screen** (`app/household/[id].tsx`):
- Household name (editable)
- Managed meal slots, shopping days, batch cook days (moved here from per-user prefs)
- Members list (display names)
- "Share invite link" button
- "Leave household" (destructive; if last member → household is deleted)

### Meal Plan Tab

Household name shown in header subtitle (e.g. "Our Family" under "Meal Plan").

### No Changes

Shopping, Recipes, individual meal/recipe screens — all work transparently via updated hooks.

## New Edge Functions

| Function | Purpose |
|---|---|
| `create-household` | Create household + add creator as owner |
| `create-invite` | Generate invite token for a household |
| `join-household` | Validate token + add user to household |

## TypeScript Types (new)

```typescript
export type HouseholdRole = 'owner' | 'member';

export interface Household {
  id: string;
  name: string;
  createdBy: string;
  managedMealSlots: MealSlot[];
  shoppingDays: number[];
  batchCookDays: number;
  createdAt: string;
  updatedAt: string;
}

export interface HouseholdMember {
  id: string;
  householdId: string;
  userId: string;
  displayName: string | null;  // joined from profiles
  role: HouseholdRole;
  joinedAt: string;
}

export interface HouseholdInvite {
  id: string;
  householdId: string;
  token: string;
  createdBy: string;
  expiresAt: string;
  createdAt: string;
}
```
